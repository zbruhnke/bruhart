# Bru-Hart 300-Page SEO Growth Plan

Date: April 26, 2026

Scope: Bru-Hart Industries, current `codex/seo-serp-content-plan` branch plus Google Search Console exports from April 26, 2026.

## Position

Bru-Hart does not need to wait for mature Google Search Console data before expanding the site architecture. The current GSC exports are too small to constrain strategy:

| Export Signal | Value |
| --- | ---: |
| Clicks | 1 |
| Impressions | 2 |
| Query rows | 0 |
| Page rows with impressions | 6 |
| Time range | Last 24 hours |

The only defensible reading is that Google has barely started seeing/testing the site. That means GSC should be used later for prioritization, cannibalization checks, and query refinement, not as a reason to delay obvious page architecture.

The real constraint is not whether Bru-Hart can get to 300 URLs. It can. The constraint is whether those URLs are useful enough to index, rank, convert, and protect the brand. A 300-page site built from thin templated copy will create index bloat. A 300-page site built from real products, real service areas, real photos, real case studies, manufacturer support, and the owner's industry reputation can become a serious asset.

See `docs/best-in-market-content-audit.md` and `docs/seo-url-quality-benchmark.md` for the per-URL competitor quality review. The short version: Bru-Hart can beat many competitor URLs with better expert guidance, RFQ checklists, sourcing transparency, and proof, but Master Halco's best pages currently beat most generated Bru-Hart pages on specs, drawings, downloads, warranties, and product/application depth.

## Why 300 Pages Is Achievable

Three things make 300 pages realistic in the short term:

| Asset | Why It Matters |
| --- | --- |
| Real product breadth | Bru-Hart can cover wholesale fence materials, agricultural fence, gates, access control, high-security gates/barriers, and custom components. |

| Real reputation | A known industry veteran can produce testimonials, project stories, buyer advice, field notes, and credibility content competitors cannot fake. |

The site should not wait for GSC to validate obvious categories like chain link, wood fence materials, farm gates, field fence, gate operators, pipe/tube, access control, crash-rated gates, or Branford fence supply. Those are core market surfaces.

The site should wait before mass-publishing dozens of near-duplicate city pages. Local pages should be based on real service areas, real product fit, real photos, delivery/service reality, and differentiated copy.

## Manufacturer And Brand Pages

Target count: 35.

Do not publish manufacturer pages unless Bru-Hart can legally and commercially support the relationship, trademarks, product availability, and approved language.

Manufacturer reputation should be treated as a conversion layer. If a buyer recognizes the brand, the page should quickly answer: what Bru-Hart carries, what the brand is known for, which product lines fit which jobs, how to request a quote, and why Bru-Hart chose to support that manufacturer. The message is not "we carry everything." The message is "we are selective, we know the category, and we do not sell junk."

Recommended page types:

| Page Type | Example |
| --- | --- |
| Partner overview | `/manufacturers/[brand]` |
| Brand category support | `/manufacturers/[brand]/gate-operators` |
| Compatible components | `/manufacturers/[brand]/replacement-parts` |
| Product line explainer | `/manufacturers/[brand]/access-control` |
| Quote/support page | `/manufacturers/[brand]-dealer-distributor-support` |

Recommended manufacturer page modules:

| Module | Purpose |
| --- | --- |
| What they are known for | Lets strong brands speak for themselves without empty praise. |
| What Bru-Hart carries | Prevents overpromising and improves quote quality. |
| Best-fit applications | Connects the manufacturer to products, industries, and local use cases. |
| Why Bru-Hart carries them | Makes selectivity explicit: reliability, parts support, field reputation, availability, or spec fit. |
| Related products | Routes brand demand into Bru-Hart's product/category architecture. |
| RFQ checklist | Captures model, size, finish, rating, operator, hardware, or replacement-part details. |

Manufacturer page guardrails:

| Risk | Guardrail |
| --- | --- |
| Unauthorized dealer implication | Use only accurate relationship language approved by the business. |
| Trademark misuse | Use brand names factually and avoid rewriting manufacturer claims as Bru-Hart claims. |
| Inventory mismatch | Mark product lines as stocked, sourced, special-order, or quote-only. |
| Duplicate manufacturer copy | Write from Bru-Hart's buyer-support perspective, not copied catalog text. |
| Overclaiming ratings | Keep crash, safety, and certification language tied to specific documented product configurations. |

## Why Not Publish 300 Pages Tomorrow

The reason to avoid publishing 300 pages immediately is not that Google needs more data first. It is that Bru-Hart needs enough unique proof, product truth, internal linking, and conversion logic to keep those pages from becoming thin.

Publish fast, but not careless:

| Risk | How To Control It |
| --- | --- |
| Thin content | Require unique product/use-case/local proof per page. |
| Cannibalization | Define one primary intent per URL before writing. |
| Overclaiming | Keep crash-rated vs crash-tested language precise. |
| Local spam | Only create city/service pages with real service fit and differentiated details. |
| Inventory mismatch | Mark pages as stocked, special-order, sourced, or quote-only. |
| Poor conversion | Every page should route to RFQ, phone, related products, and proof. |

## Technical Recommendation

The current branch uses a single TypeScript content registry. That is fine for the first 20 pages, but it should not become a 300-page monolith.

Before scaling past roughly 75-100 pages, move to a data-driven structure:

| Layer | Recommendation |
| --- | --- |
| Content source | Sanity, MDX, or structured JSON/YAML collections by page family. |
| Templates | Product, local service, industry, manufacturer, case study, resource, testimonial. |
| Routing | Keep dynamic routes, generate metadata and sitemap from content records. |
| Governance | Add fields for page status, page owner, proof assets, product availability, and last reviewed date. |
| Launch control | Allow draft/preview, noindex, and staged publish. |

This lets Bru-Hart build hundreds of pages without hand-maintaining hundreds of route files or giant TS objects.
